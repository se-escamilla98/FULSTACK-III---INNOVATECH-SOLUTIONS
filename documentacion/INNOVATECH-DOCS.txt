--------------------------------------------------------------------------------
1.2 ARQUITECTURA DEL SISTEMA
--------------------------------------------------------------------------------

El sistema implementa una arquitectura de microservicios con las siguientes
capas en cada servicio:

  Capa de Entrada (App/Routes)
    Recibe peticiones HTTP externas. Define las rutas y las conecta
    con los controladores.

  Capa de Controlador
    Valida que los datos de la peticion sean correctos (campos requeridos,
    tipos). No contiene logica de negocio.

  Capa de Servicio
    Donde reside la logica de negocio (reglas y calculos). Es la unica
    que interactua con Prisma y la base de datos.

  Capa de Persistencia (Prisma ORM)
    Transforma filas de la base de datos en objetos TypeScript y viceversa.
    Genera clientes con autocompletado y validacion de tipos.

COMUNICACION ENTRE SERVICIOS (v6.0 con Traefik y Swarm):

  [Navegador / Cliente externo]
       |
       v  puerto 80
  [Traefik v3.0]  <- Load Balancer, Health Check, Reverse Proxy
       |
       v  round-robin entre 3 instancias
  [BFF Gateway x3 replicas]  <- Swagger Unificado, JWT, Circuit Breaker, /whoami
       |
       +---> [MS-Projects] ---> [db-projects]
       |
       +---> [MS-Tasks   ] ---> [db-tasks   ]
       |
       +---> [MS-Teams   ] ---> [db-teams   ]

  [MS-Monitor :4000]  <- Dashboard HTML + JSON, consulta /health de cada servicio

  [Docker Swarm]  <- Mantiene replicas, auto-healing, rolling updates

NOTAS DE ARQUITECTURA:
  - Los microservicios NO tienen puertos expuestos al host (Zero Trust a nivel red).
  - Traefik es el unico punto de entrada a la API (puerto 80).
  - El BFF corre con 3 replicas en Swarm; Traefik balancea via DNS round-robin.
  - MS-Monitor es un servicio independiente que consulta /health de cada MS.
  - El frontend corre en puerto 5173 y apunta a http://localhost (Traefik).
  - Docker Swarm gestiona el reinicio automatico si cae una replica.

--------------------------------------------------------------------------------
1.3 JUSTIFICACION TECNICA (PARA DEFENSA ORAL / INFORME EFT)
--------------------------------------------------------------------------------

MICROSERVICIO:
  Componente de software independiente que se encarga de un dominio especifico
  (ej: Tareas, Proyectos, Equipos). Permite escalar horizontalmente cada modulo
  de forma independiente segun la demanda del negocio.

ENDPOINT:
  Punto de acceso o URL de una API donde se envia una peticion HTTP.

PAYLOAD:
  Los datos reales que se envian en el cuerpo de una peticion JSON.

MIGRACION (PRISMA):
  Proceso que traduce los cambios del modelo de datos en el codigo a cambios
  estructurales en la base de datos SQL (crea o modifica tablas).

MIDDLEWARE:
  Codigo que se ejecuta entre la recepcion de la peticion y la respuesta final.
  Usado en este proyecto para validar JWT en cada endpoint (Zero Trust).

JWT (JSON Web Token):
  Estandar para transmitir informacion de forma segura entre partes como un
  objeto JSON firmado. En Innovatech, cada peticion debe incluir un token valido.

CIRCUIT BREAKER:
  Patron de resiliencia que protege al sistema cuando un microservicio falla
  repetidamente. Funciona igual que un disyuntor electrico: cuando hay demasiados
  fallos, "abre el circuito" y retorna un fallback inmediato en vez de esperar
  timeouts. Implementado con la libreria Opossum en el BFF Gateway.

  Estados del circuito:
    CLOSED    -> Todo normal. Las llamadas pasan al microservicio.
    OPEN      -> Demasiados fallos. Las llamadas retornan el fallback
                 inmediatamente sin tocar el microservicio caido.
    HALF-OPEN -> Periodo de prueba. Se deja pasar una llamada para verificar
                 si el servicio se recupero. Si pasa: vuelve a CLOSED.
                 Si falla: vuelve a OPEN.

  Configuracion en Innovatech:
    timeout: 3000ms               <- llamadas que tardan mas de 3s cuentan como fallo
    errorThresholdPercentage: 50  <- si el 50% de llamadas fallan, abre el circuito
    resetTimeout: 10000ms         <- despues de 10s en Open, pasa a Half-Open

  Principio "Fail Fast": en vez de acumular threads bloqueados esperando timeouts,
  el circuito abierto retorna inmediatamente. Protege al BFF, al microservicio
  caido (no lo bombardea) y al usuario (recibe respuesta util de inmediato).

ZERO TRUST:
  Modelo de seguridad que exige autenticacion en CADA peticion, sin importar
  el origen. En Innovatech, cada microservicio valida el JWT individualmente,
  incluso en acceso directo (sin pasar por el BFF).

TRAEFIK (REVERSE PROXY / LOAD BALANCER):
  Software que recibe todo el trafico externo y lo distribuye entre las instancias
  disponibles de un servicio. En Innovatech, Traefik escucha en el puerto 80 y
  reenvía las peticiones al BFF Gateway usando DNS round-robin.
  Ventaja: permite escalar el BFF sin cambiar la URL del cliente.
  Proveedor de configuracion: file provider (traefik/dynamic.yml), ya que el
  Docker provider no funciona en Docker Desktop sobre Windows.

DOCKER SWARM:
  Modo de cluster nativo de Docker que convierte un host en un nodo manager
  capaz de gestionar servicios con replicas, restart automatico y rolling updates.
  Diferencia con Compose: Compose levanta contenedores; Swarm los gestiona como
  servicios con N replicas y garantiza que siempre haya el numero configurado.

  Terminos clave de Swarm:
    Stack    -> grupo de servicios desplegados juntos (equivale al compose)
    Service  -> unidad gestionada con replicas (ej: innovatech_bff con 3 replicas)
    Task     -> una instancia concreta de un service (un contenedor)
    Overlay  -> red virtual que conecta contenedores en el cluster
    Config   -> almacen de archivos de configuracion dentro del Swarm

AUTO-HEALING:
  Cuando un contenedor (task) falla, el manager de Swarm detecta que el numero
  de replicas activas es menor al deseado y lanza una nueva instancia
  automaticamente en segundos, sin intervencion humana.

ROLLING UPDATE (ACTUALIZACION SIN DOWNTIME):
  Al actualizar una imagen, Swarm reemplaza las replicas de a una: primero levanta
  la nueva, verifica que este saludable, luego apaga la antigua.
  Configurado con: parallelism=1, delay=10s, order=start-first.

HEALTHCHECK:
  Instruccion en el Dockerfile que permite a Docker (y Swarm) saber si un
  contenedor esta realmente listo para recibir trafico. Usa wget contra /health.
  Si falla 3 veces seguidas, Docker marca el contenedor como "unhealthy".

MS-MONITOR:
  Microservicio propio que consulta el endpoint /health de cada servicio cada
  15 segundos y expone un dashboard HTML en el puerto 4000. Permite visualizar
  el estado del sistema sin acceder a logs de Docker.

JUSTIFICACION DE ELECCIONES TECNOLOGICAS:
  "La eleccion de TypeScript y Prisma responde a la necesidad de Innovatech de
  contar con un sistema mantenible y con baja tasa de errores en produccion.
  La arquitectura de microservicios sobre Docker Swarm garantiza que la solucion
  pueda escalar horizontalmente con auto-healing, permitiendo que cada modulo
  crezca de forma independiente segun la demanda del negocio. Traefik como
  load balancer elimina el punto unico de fallo en el BFF Gateway."

--------------------------------------------------------------------------------

MS_TASKS_URL=http://localhost:3001
MS_PROJECTS_URL=http://localhost:3002
MS_TEAMS_URL=http://localhost:3003
JWT_SECRET=innovatech-secret-key-2026

--------------------------------------------------------------------------------
2.6 DOCKERFILES (con HEALTHCHECK para Swarm)
--------------------------------------------------------------------------------

El HEALTHCHECK permite que Docker Swarm sepa si un contenedor esta listo para
recibir trafico. Si falla 3 veces, Swarm lo marca "unhealthy" y lanza uno nuevo.

--- Dockerfile para ms-tasks (puerto 3001) ---
FROM node:22-alpine
RUN apk add --no-cache openssl
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npx prisma generate
RUN npm run build
EXPOSE 3001
HEALTHCHECK --interval=15s --timeout=5s --start-period=30s --retries=3 \
  CMD wget -q -O- http://localhost:3001/health || exit 1
CMD ["node", "dist/app.js"]

--- Dockerfile para ms-projects (puerto 3002) ---
  (igual al anterior cambiando EXPOSE y HEALTHCHECK a puerto 3002)

--- Dockerfile para ms-teams (puerto 3003) ---
  (igual al anterior cambiando EXPOSE y HEALTHCHECK a puerto 3003)

--- Dockerfile para bff-gateway (NO usa Prisma) ---
FROM node:22-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install --include=dev
COPY . .
RUN npm run build
EXPOSE 3000
HEALTHCHECK --interval=15s --timeout=5s --start-period=30s --retries=3 \
  CMD wget -q -O- http://localhost:3000/health || exit 1
CMD ["node", "dist/app.js"]

--- Dockerfile para ms-monitor (puerto 4000) ---
FROM node:22-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build
EXPOSE 4000
CMD ["node", "dist/app.js"]

NOTAS CRITICAS:
- "RUN apk add --no-cache openssl" es OBLIGATORIO en microservicios con Prisma.
  Sin esta linea ocurre el error "libssl.so.1.1: No such file or directory".
- El BFF usa "npm install --include=dev" para incluir TypeScript dentro de Docker.
- El BFF NO necesita openssl ni prisma generate.
- HEALTHCHECK usa wget (disponible en alpine). El --start-period=30s da tiempo
  para que el servicio y la BD arranquen antes de empezar a evaluar la salud.
- Swarm solo reemplaza tareas "unhealthy"; una tarea que "falla" (exit code != 0)
  se reinicia por restart_policy, independiente del HEALTHCHECK.

--------------------------------------------------------------------------------
2.7 ARCHIVO .dockerignore (OBLIGATORIO EN TODOS)
--------------------------------------------------------------------------------

Sin este archivo Docker copia el node_modules local al contenedor y pisa la
instalacion de dependencias, causando "sh: tsc: not found" y
"sh: prisma: not found" aunque esten declarados en el package.json.

Crear .dockerignore en la raiz de CADA microservicio y el BFF con este contenido:

  node_modules
  dist
  .env

--------------------------------------------------------------------------------
2.12 TRAEFIK - LOAD BALANCER (traefik/dynamic.yml)
--------------------------------------------------------------------------------

Traefik usa "file provider": lee su configuracion de un archivo estatico en vez
de preguntar al Docker daemon. Esto es necesario en Windows Docker Desktop
porque el Docker socket no es accesible desde contenedores.

Archivo traefik/dynamic.yml:
  http:
    routers:
      bff:
        rule: "PathPrefix(`/`)"
        entryPoints:
          - web
        service: bff-service
    services:
      bff-service:
        loadBalancer:
          servers:
            - url: "http://bff:3000"
          healthCheck:
            path: /health
            interval: "10s"
            timeout: "3s"

Explicacion:
  - Todo el trafico que llega al puerto 80 se envia al servicio bff-service.
  - bff-service apunta a http://bff:3000 (DNS interno de Docker).
  - En Swarm, "bff" resuelve al VIP del servicio, que distribuye entre replicas.
  - healthCheck verifica /health cada 10s; si falla, Traefik deja de enviar
    trafico a esa instancia.

En docker-compose.yml, el volumen de Traefik:
  volumes:
    - ./traefik/dynamic.yml:/etc/traefik/dynamic.yml:ro

En docker-stack.yml, Traefik usa Docker Config (el archivo se sube al Swarm):
  configs:
    - source: traefik-dynamic
      target: /etc/traefik/dynamic.yml

2.16 EJECUTAR MIGRACIONES
--------------------------------------------------------------------------------

Las migraciones crean las tablas en PostgreSQL segun el schema.prisma.
REQUISITO: Las bases de datos deben estar corriendo antes de migrar.
CRITICO: Sin migraciones el sistema falla con "table does not exist" o
         "socket hang up" cuando el BFF intenta consultar datos.

Con Docker Compose (BD expuesta en localhost:5433/5434/5435):
  cd ms-tasks       && npx prisma migrate dev --name init
  cd ../ms-projects && npx prisma migrate dev --name init
  cd ../ms-teams    && npx prisma migrate dev --name init

Con Docker Swarm (BD NO expuesta al host, ejecutar dentro del contenedor):
  docker exec -it $(docker ps -qf name=innovatech_ms-tasks) npx prisma migrate dev --name init
  docker exec -it $(docker ps -qf name=innovatech_ms-projects) npx prisma migrate dev --name init
  docker exec -it $(docker ps -qf name=innovatech_ms-teams) npx prisma migrate dev --name init

Resultado esperado para cada migracion:
  Prisma schema loaded from prisma\schema.prisma
  Applying migration `20260518_init`
  Database changes applied.

DIFERENCIA ENTRE GENERATE Y MIGRATE:
  npx prisma generate   -> Crea los tipos TypeScript desde el schema.
                           Necesario para que el codigo compile.
  npx prisma migrate    -> Crea o actualiza las tablas en PostgreSQL.
                           Necesario para que los datos se guarden.
  Ambos son obligatorios. Si solo haces generate sin migrate, el
  codigo compila pero falla al intentar acceder a la base de datos.

2.19 ERRORES COMUNES Y SOLUCIONES
--------------------------------------------------------------------------------

ERROR: sh: tsc: not found / sh: prisma: not found (Docker)
  Causa:    Falta .dockerignore. Docker copia node_modules local y pisa la
            instalacion del contenedor.
  Solucion: Crear .dockerignore en cada servicio con: node_modules / dist / .env
            Luego ejecutar: docker compose build --no-cache

ERROR: Authentication failed against database server
  Causa:    Credenciales del .env no coinciden con las del docker-compose.yml.
  Solucion: Verificar .env contra la tabla de credenciales de la seccion 2.4.

ERROR: Environment variable not found: DATABASE_URL
  Causa:    Faltan comillas dobles en el .env.
  Solucion: DATABASE_URL="postgresql://..." (comillas dobles OBLIGATORIAS)

ERROR: The table 'public.Project' does not exist
  Causa:    Migraciones no ejecutadas en esta sesion.
  Solucion: Ejecutar las tres migraciones de la seccion 2.13.

ERROR: libssl.so.1.1 not found (Docker)
  Causa:    node:22-alpine no incluye OpenSSL que Prisma necesita.
  Solucion: Agregar al Dockerfile de cada microservicio con Prisma:
    RUN apk add --no-cache openssl

ERROR: Bind for 0.0.0.0:5435 failed: port is already allocated
  Causa:    Un contenedor Docker de OTRO proyecto ocupa el puerto.
  Solucion: docker ps -> identificar el contenedor conflictivo -> docker stop ID

ERROR: socket hang up (en BFF)
  Causa:    El BFF llega al microservicio pero la BD no tiene tablas (migraciones
            pendientes) o el JWT_SECRET no esta configurado en el microservicio.
  Solucion: Ejecutar migraciones y verificar JWT_SECRET en docker-compose.yml.

ERROR: Request failed with status code 401 (desde BFF a microservicio)
  Causa:    El BFF no esta reenviando el token JWT en las llamadas axios.
  Solucion: Verificar que el authHeaders helper esta siendo usado en cada
            llamada axios de los services del BFF.

ERROR: Cannot find module '../services/...'
  Causa:    Nombres de carpetas no coinciden con los imports.
  Solucion: Verificar que las carpetas se llaman services (plural), factories, etc.

ERROR: ECMAScript imports cannot be written in CommonJS file
  Causa:    tsconfig.json tiene configuracion incorrecta.
  Solucion: En tsconfig.json usar "module": "commonjs" y "target": "es2020".
            Eliminar "verbatimModuleSyntax" completamente.

ERROR: GET /tasks/:id atrapa /tasks/project/:projectId
  Causa:    Orden incorrecto de rutas en app.ts.
  Solucion: La ruta especifica SIEMPRE debe ir antes de la generica:
    app.get('/tasks/project/:projectId', ...)  <- primero
    app.get('/tasks/:id', ...)                 <- despues

ERROR: No se puede marcar como finalizado (sigue fallando)
  Causa:    ms-projects usa 'http://localhost:3001' dentro de Docker.
            Localhost apunta al propio contenedor, no a ms-tasks.
  Solucion: Usar la variable de entorno MS_TASKS_URL en project.service.ts:
    const MS_TASKS_URL = process.env.MS_TASKS_URL || 'http://ms-tasks:3001';

3.3 COMO USAR CMD (curl.exe)
--------------------------------------------------------------------------------

En Windows SIEMPRE usar "curl.exe" (no "curl"). PowerShell intercepta "curl"
y lo convierte en Invoke-WebRequest con sintaxis diferente.

GET:
  curl.exe -H "Authorization: Bearer TOKEN" URL

POST:
  curl.exe -X POST URL -H "Content-Type: application/json" -H "Authorization: Bearer TOKEN" -d "{\"key\": \"value\"}"

PATCH:
  curl.exe -X PATCH URL -H "Content-Type: application/json" -H "Authorization: Bearer TOKEN" -d "{\"key\": \"value\"}"

DELETE:
  curl.exe -X DELETE -H "Authorization: Bearer TOKEN" URL

Nota: En CMD las comillas dentro del body se escapan con backslash: \"

--------------------------------------------------------------------------------
3.4 COMO USAR POWERSHELL (Invoke-WebRequest)
--------------------------------------------------------------------------------

Usar siempre -UseBasicParsing para evitar advertencias de seguridad.
El body usa comillas simples afuera y dobles adentro.

GET:
  Invoke-WebRequest -Uri "URL" -Method GET -Headers @{Authorization="Bearer TOKEN"} -UseBasicParsing

POST:
  Invoke-WebRequest -Uri "URL" -Method POST -ContentType "application/json" -Headers @{Authorization="Bearer TOKEN"} -Body '{"key": "value"}' -UseBasicParsing

PATCH:
  Invoke-WebRequest -Uri "URL" -Method PATCH -ContentType "application/json" -Headers @{Authorization="Bearer TOKEN"} -Body '{"key": "value"}' -UseBasicParsing

DELETE:
  Invoke-WebRequest -Uri "URL" -Method DELETE -Headers @{Authorization="Bearer TOKEN"} -UseBasicParsing

TRUCO: Guardar el token en variable para no copiarlo manualmente cada vez:
  $response = Invoke-WebRequest -Uri "http://localhost:3000/auth/login" -Method POST -ContentType "application/json" -Body '{"apiKey": "admin-key-innovatech"}' -UseBasicParsing
  $token = ($response.Content | ConvertFrom-Json).token
  (usar $token en los comandos siguientes)

3.5 AUTENTICACION JWT - GENERAR Y USAR TOKENS

El sistema implementa Zero Trust: TODOS los endpoints requieren JWT valido,
tanto en el BFF (puerto 3000) como en cada microservicio directamente.
La unica excepcion es POST /auth/login.

API Keys disponibles:
  "admin-key-innovatech"  -> rol admin  (GET, POST, PATCH, DELETE)
  "reader-key-innovatech" -> rol reader (solo GET)

--- GENERAR TOKEN (POST /auth/login) ---

CMD:

PowerShell (guardando en variable):
  $response = Invoke-WebRequest -Uri "http://localhost:3000/auth/login" -Method POST -ContentType "application/json" -Body '{"apiKey": "admin-key-innovatech"}' -UseBasicParsing

3.12 PRUEBAS DEL CIRCUIT BREAKER
--------------------------------------------------------------------------------

NOTA: El Circuit Breaker vive en el BFF. Al usar Docker Swarm con 3 replicas
del BFF, cada replica tiene su propio circuito. Traefik puede enrutar a una
replica diferente en cada peticion, por lo que el estado del circuito puede
variar entre replicas. Para demostrar el circuit breaker de forma predecible,
usar Docker Compose con una sola instancia del BFF.

--- PRUEBA 1: ESTADO NORMAL (CIRCUITO CLOSED) ---

Con todos los servicios corriendo, el circuito esta en CLOSED.

PowerShell:
  Invoke-WebRequest -Uri "http://localhost/api/projects" -Headers @{Authorization="Bearer $token"} -UseBasicParsing
Resultado esperado (200): [] o lista real de proyectos.

--- PRUEBA 2: SIMULAR CAIDA DE MS-TASKS (ACTIVAR CIRCUIT BREAKER) ---

Con Docker Compose:
  PASO 1: Detener ms-tasks:
    docker compose stop ms-tasks

Con Docker Swarm:
  PASO 1: Escalar ms-tasks a 0:
    docker service scale innovatech_ms-tasks=0

PASO 2: Llamar a endpoint de tareas via BFF (primera llamada puede tardar 3s):
  CMD:      curl.exe -H "Authorization: Bearer TOKEN" http://localhost/api/tasks/UUID-CUALQUIERA
  PowerShell: Invoke-WebRequest -Uri "http://localhost/api/tasks/UUID-CUALQUIERA" -Headers @{Authorization="Bearer $token"} -UseBasicParsing

Resultado esperado (200 OK con fallback):
  {"error": "MS-Tasks no disponible. Intente mas tarde."}

NOTA: El circuit breaker retorna HTTP 200 con mensaje de fallback, NO un error 503.
El BFF sigue respondiendo de forma controlada.

PASO 3: Hacer mas llamadas -> las siguientes retornan el fallback INMEDIATAMENTE
sin esperar el timeout de 3s. Esto confirma que el circuito esta en estado OPEN.

--- PRUEBA 3: FALLBACK DE LISTA VACIA ---

Con Docker Compose:   docker compose stop ms-projects
Con Docker Swarm:     docker service scale innovatech_ms-projects=0

Llamar a getAllProjects via BFF:
  CMD:      curl.exe -H "Authorization: Bearer TOKEN" http://localhost/api/projects

Resultado esperado (200 OK):
  []  <- lista vacia, NO un error 500

Para listados, el fallback retorna [] en vez de romper el frontend con un error.

--- PRUEBA 4: RECUPERACION (HALF-OPEN -> CLOSED) ---

PASO 1: Con ms-tasks detenido, esperar 10 segundos (resetTimeout del circuito).
PASO 2: Volver a levantar ms-tasks:
  Docker Compose:  docker compose start ms-tasks
  Docker Swarm:    docker service scale innovatech_ms-tasks=1
PASO 3: Esperar 5 segundos para que ms-tasks inicialice.
PASO 4: Llamar al endpoint -> el circuito pasa de HALF-OPEN a CLOSED.

3.13 PRUEBA DE AUTO-HEALING CON DOCKER SWARM
--------------------------------------------------------------------------------

Auto-healing es la capacidad de Swarm de detectar que una replica fallo y
lanzar una nueva automaticamente sin intervencion humana.

REQUISITO: Sistema corriendo con docker stack deploy (Swarm activo).

PASO 1: Verificar que hay 3 replicas del BFF activas:
  docker service ps innovatech_bff --format "table {{.ID}}\t{{.Name}}\t{{.CurrentState}}"

  Resultado esperado:
    ID             NAME               CURRENT STATE
    abc123...      innovatech_bff.1   Running X minutes ago
    def456...      innovatech_bff.2   Running X minutes ago
    ghi789...      innovatech_bff.3   Running X minutes ago

PASO 2: Identificar el container ID de una replica y matarla:
  $containerId = docker ps --filter "name=innovatech_bff" --format "{{.ID}}" | Select-Object -First 1
  docker stop $containerId

PASO 3: Esperar 10 segundos y verificar que Swarm lanzo una nueva:
  Start-Sleep -Seconds 10
  docker service ps innovatech_bff --format "table {{.ID}}\t{{.Name}}\t{{.CurrentState}}\t{{.Error}}"

  Resultado esperado:
    ID          NAME                   CURRENT STATE             ERROR
    xyz999...   innovatech_bff.1       Running 8 seconds ago     <- NUEVA replica
    abc123...    \_ innovatech_bff.1   Failed 18 seconds ago     "task: non-zero exit (137)"
    def456...   innovatech_bff.2       Running X minutes ago
    ghi789...   innovatech_bff.3       Running X minutes ago

  El exit code 137 = SIGKILL (docker stop). Swarm detecto la falla y lanzo la
  replica nueva en menos de 10 segundos. En todo momento el sistema mantuvo
  al menos 2/3 replicas activas, por lo que el servicio nunca estuvo caido.

PASO 4: Verificar que el sistema sigue respondiendo durante la prueba:
  curl.exe http://localhost/health
  Resultado esperado: {"status":"ok","service":"bff-gateway",...}

RESUMEN DE LA PRUEBA:
  Antes del kill: 3/3 replicas Running
  Durante kill:   2/3 replicas Running (temporal, ~segundos)
  Despues:        3/3 replicas Running (Swarm repuso la replica automaticamente)

ERROR 1: Traefik no puede acceder al Docker socket en Windows Docker Desktop
--------------------------------------------------------------------------------

CUANDO OCURRIO:
  Al intentar configurar Traefik con el provider "docker" para que detectara
  los contenedores automaticamente.

MENSAJE DE ERROR:
  ERR Failed to retrieve information of the docker client and server host
  error="Error response from daemon: "

CAUSA RAIZ:
  Traefik con Docker provider necesita acceder al socket de Docker
  (/var/run/docker.sock en Linux) para consultar los contenedores activos.
  En Docker Desktop para Windows, ese socket no esta en la ruta estandar de Linux
  y el daemon de Docker no escucha en la interfaz de red del contenedor.

INTENTOS FALLIDOS:
  Intento 1: Montar /var/run/docker.sock directamente
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock
    Resultado: "Error response from daemon" - el socket no existe en esa ruta
    dentro del contexto de WSL2.

  Intento 2: Usar TCP tcp://host.docker.internal:2375
    command:
      - "--providers.docker.endpoint=tcp://host.docker.internal:2375"
    Resultado: Docker Desktop limita la interfaz TCP a 127.0.0.1 del host,
    que no es accesible desde dentro de un contenedor.

  Intento 3: Socket proxy (tecnativa/docker-socket-proxy)
    Resultado: La imagen hace peticiones a GET /v1.24/version y Docker Desktop
    responde 400 porque espera API version >= 1.41. El proxy no tiene opcion
    para cambiar la version negociada.

  Intento 4: Variable DOCKER_API_VERSION=1.41
    Resultado: La variable solo afecta al cliente CLI, no al proxy interno de
    la imagen del socket-proxy.

SOLUCION DEFINITIVA:
  Cambiar a Traefik file provider. En lugar de que Traefik consulte la API
  de Docker, se le da un archivo de configuracion estatico:

  traefik/dynamic.yml:
    http:
      routers:
        bff:
          rule: "PathPrefix(`/`)"
          entryPoints: [web]
          service: bff-service
      services:
        bff-service:
          loadBalancer:
            servers:
              - url: "http://bff:3000"
            healthCheck:
              path: /health
              interval: "10s"
              timeout: "3s"

  Y en docker-compose.yml:
    command:
      - "--providers.file.filename=/etc/traefik/dynamic.yml"
    volumes:
      - ./traefik/dynamic.yml:/etc/traefik/dynamic.yml:ro

POR QUE FUNCIONA:
  Traefik lee el archivo en lugar de preguntar al daemon Docker. El nombre
  "bff" resuelve via el DNS interno de Docker (que si funciona).

LECCION:
  El Docker provider de Traefik es una limitacion arquitectonica de Docker
  Desktop en Windows, no un bug de Traefik. El file provider es una alternativa
  igualmente valida y usada en produccion.

ERROR 2: El watcher de Traefik no detecta cambios en Windows/WSL2
--------------------------------------------------------------------------------

CUANDO OCURRIO:
  Al modificar traefik/dynamic.yml para agregar 3 servidores con IPs explicitas
  y verificar que Traefik recargara automaticamente (--providers.file.watch=true).

COMPORTAMIENTO:
  El archivo se modifico. Traefik NO recargo. La API de Traefik en
  http://localhost:8080/api/http/services seguia mostrando la configuracion
  anterior con un solo servidor.

CAUSA RAIZ:
  Traefik usa el mecanismo inotify de Linux para detectar cambios en archivos.
  Cuando el archivo dynamic.yml esta en el sistema de archivos de Windows y
  se monta como volumen en Docker (que corre dentro de WSL2), los cambios
  hechos desde Windows no generan eventos inotify dentro del contenedor Linux.
  Es un problema conocido de la capa de virtualizacion Windows -> WSL2 -> Linux.

SOLUCION:
  Reiniciar el contenedor de Traefik despues de cada cambio al dynamic.yml:
    docker compose restart traefik
  O en Swarm actualizar el config y redesplegar el servicio:
    docker config rm traefik-dynamic
    docker config create traefik-dynamic ./traefik/dynamic.yml
    docker service update --force innovatech_traefik

LECCION:
  En produccion real (Linux nativo), el watch=true funciona perfectamente.
  En Windows es un workaround necesario.

ERROR 3: Login retorna 400 - formato de credenciales incorrecto
--------------------------------------------------------------------------------

CUANDO OCURRIO:
  Al intentar obtener token JWT con el body {"apiKey": "admin-key-innovatech"}
  y recibir {"error": "Usuario y contraseña son requeridos."}.

CAUSA RAIZ:
  La implementacion de autenticacion fue cambiada durante el desarrollo.
  La version original usaba API Keys estaticas (apiKey).
  La version actual usa username + password con bcrypt (mas segura).

FORMATO INCORRECTO (version anterior):
  POST /auth/login
  {"apiKey": "admin-key-innovatech"}

FORMATO CORRECTO (version actual con bcrypt):
  POST /auth/login
  {"username": "admin", "password": "admin123"}

USUARIOS DISPONIBLES (en bff-gateway/src/auth/auth.routes.ts):
  admin     / admin123   -> rol admin  (todos los metodos)
  sebastian / duoc2026   -> rol admin
  lector    / lector123  -> rol reader (solo GET)

POR QUE BCRYPT:
  bcrypt es un algoritmo de hashing de passwords disenado para ser lento
  (por diseno), lo que lo hace resistente a ataques de fuerza bruta.
  Incluso si alguien roba la base de datos con los hashes, no puede recuperar
  la contrasena original facilmente.
  En bff-gateway usa bcrypt.hashSync al inicializar el modulo (una sola vez)
  y bcrypt.compare en cada login. Nunca se almacena la contrasena en texto plano.

LECCION:
  Siempre documentar el formato de autenticacion en detalle. Cuando cambia
  la implementacion, actualizar toda la documentacion de pruebas.

ERROR 4: Los nombres bff-1, bff-2, bff-3 no resuelven en Docker Compose
--------------------------------------------------------------------------------

CUANDO OCURRIO:
  Al intentar poner 3 entradas explicitas en traefik/dynamic.yml para que
  Traefik hiciera round-robin real entre las 3 instancias del BFF.

CONFIGURACION ERRONEA PROBADA:
  services:
    bff-service:
      loadBalancer:
        servers:
          - url: "http://bff-1:3000"
          - url: "http://bff-2:3000"
          - url: "http://bff-3:3000"

MENSAJE DE ERROR (dentro del contenedor Traefik):
  wget: bad address 'bff-1:3000'
  wget: bad address 'bff-2:3000'

CAUSA RAIZ:
  En Docker Compose v2 con --scale bff=3, los contenedores se llaman
  internamente:
    fulstack-iii---innovatech-solutions-1-bff-1
    fulstack-iii---innovatech-solutions-1-bff-2
    fulstack-iii---innovatech-solutions-1-bff-3
  El DNS interno de Docker solo registra el NOMBRE DEL SERVICIO (bff) como
  alias comun que resuelve a todos, no los sufijos -1/-2/-3 sin prefijo.

SOLUCION:
  Obtener las IPs reales de los contenedores y usarlas directamente:

  # Obtener IPs
  $ip1 = docker inspect {nombre-bff-1} | ConvertFrom-Json |
    ForEach-Object { $_.NetworkSettings.Networks.PSObject.Properties.Value[0].IPAddress }
  (repetir para bff-2 y bff-3)

  # Actualizar dynamic.yml con las IPs reales:
  servers:
    - url: "http://172.18.0.9:3000"
    - url: "http://172.18.0.10:3000"
    - url: "http://172.18.0.11:3000"

  # Reiniciar Traefik para que recargue:
  docker compose restart traefik

RESULTADO VERIFICADO:
  Con las 3 IPs explicitas, los 9 requests se distribuyeron entre
  3 hostnames distintos (confirmado con el endpoint /whoami).

LIMITACION:
  Las IPs de contenedores cambian cada vez que se recrea el stack.
  Esta solucion es valida para demostrar el load balancing en una sesion
  de pruebas, pero no para produccion permanente.
  En produccion real, el Docker provider (que no funciona en Windows)
  descubriria los backends automaticamente.

ALTERNATIVA DE PRODUCCION (Linux/Kubernetes):
  Usar el Docker provider o Kubernetes Ingress, donde Traefik descubre
  los endpoints automaticamente sin IPs fijas.

--------------------------------------------------------------------------------
ERROR 5: Load balancing parece no funcionar (siempre el mismo hostname)
--------------------------------------------------------------------------------

CUANDO OCURRIO:
  Al llamar a /whoami 6 veces seguidas y ver siempre el mismo hostname,
  incluso con 3 replicas corriendo.

COMPORTAMIENTO OBSERVADO:
  Request 1 -> hostname: 680add9a75fc
  Request 2 -> hostname: 680add9a75fc
  Request 3 -> hostname: 680add9a75fc
  (siempre el mismo)

CAUSA RAIZ:
  HTTP/1.1 usa keep-alive por defecto. Cuando PowerShell hace
  Invoke-WebRequest, abre UNA conexion TCP a Traefik y la reutiliza para
  todas las peticiones siguientes. Traefik selecciona el backend UNA VEZ
  al establecer la conexion, y todas las peticiones de esa conexion van al
  mismo backend. El round-robin de Traefik actua por NUEVA CONEXION, no
  por peticion individual en la misma conexion.

POR QUE SWARM VIP TAMPOCO AYUDA:
  Docker Swarm en modo VIP (Virtual IP) crea una IP virtual unica para el
  servicio. IPVS distribuye las conexiones entrantes entre las replicas.
  Pero si Traefik tiene una conexion persistente al VIP, IPVS no la
  rebalancea: la conexion ya fue asignada a un backend y sigue ahi.

SOLUCION PARA LA DEMO:
  Configurar Traefik con 3 IPs explicitas (como en ERROR 4) para que
  Traefik abra conexiones independientes a cada backend y pueda rotar.

COMO FUNCIONA EN PRODUCCION REAL:
  En produccion, DISTINTOS CLIENTES (navegadores, apps) generan DISTINTAS
  conexiones TCP a Traefik. Cada nueva conexion se asigna al siguiente
  backend en round-robin. Un cliente real nunca nota que esta "atascado"
  en un backend porque sus peticiones son pocas y rapidas.
  La demo de /whoami desde una sola sesion de PowerShell no representa
  el comportamiento real de produccion con multiples clientes.

--------------------------------------------------------------------------------
ERROR 6: Migraciones Prisma fallan en Docker Swarm (BDs no expuestas)
--------------------------------------------------------------------------------

CUANDO OCURRIO:
  Al intentar ejecutar npx prisma migrate dev desde la maquina host en un
  stack de Swarm.

CAUSA RAIZ:
  En Docker Compose (desarrollo) las BDs tienen puertos expuestos:
    db-tasks:    5433 en el host
    db-projects: 5434 en el host
    db-teams:    5435 en el host
  Los .env de cada microservicio apuntan a localhost:5433/5434/5435.

  En Docker Swarm (produccion) las BDs NO tienen puertos expuestos al host
  (Zero Trust a nivel red). Solo son accesibles desde la red overlay interna
  del Swarm (docker_gwbridge o innovatech-overlay).

SOLUCION:
  Ejecutar las migraciones DENTRO del contenedor del microservicio,
  que si tiene acceso a la BD via la red overlay:

  # Obtener el container ID
  docker ps --filter "name=innovatech_ms-tasks" --format "{{.ID}}"

  # Ejecutar la migracion dentro del contenedor
  docker exec {container-id} npx prisma migrate deploy

  IMPORTANTE: usar "migrate deploy" (no "migrate dev") en contenedores
  de produccion. La diferencia:
    migrate dev    -> interactivo, crea archivos de migracion nuevos,
                     solo para desarrollo local
    migrate deploy -> aplica migraciones ya existentes, no interactivo,
                     ideal para contenedores y CI/CD

LECCION:
  Siempre verificar si los puertos de BD estan expuestos antes de intentar
  migrar desde el host. En Swarm, siempre usar docker exec.

--------------------------------------------------------------------------------
ERROR 7: App.tsx usaba URL hardcodeada para el login
--------------------------------------------------------------------------------

CUANDO OCURRIO:
  Al verificar que el frontend funcionara con el sistema escalado via Traefik.

CODIGO PROBLEMATICO (en frontend-web/src/App.tsx):
  const response = await axios.post('http://localhost:3000/auth/login', ...)

PROBLEMA:
  La URL apuntaba directamente al BFF en el puerto 3000, saltandose a Traefik.
  Con Traefik activo el BFF ya no expone el puerto 3000 al host, por lo que
  el login fallaba.

SOLUCION:
  Importar BFF_BASE desde bffClient.ts (que ya usaba la variable de entorno):
    import { BFF_BASE } from './api/bffClient'
    const response = await axios.post(`${BFF_BASE}/auth/login`, ...)

  Y en bffClient.ts:
    export const BFF_BASE = import.meta.env.VITE_BFF_URL ?? 'http://localhost:3000'

  En docker-compose.yml / docker-stack.yml:
    environment:
      - VITE_BFF_URL=http://localhost  (apunta a Traefik en puerto 80)

LECCION:
  Nunca hardcodear URLs de servicios internos. Siempre usar variables de entorno
  o constantes centralizadas que se puedan cambiar sin tocar el codigo.

--------------------------------------------------------------------------------
ERROR 8: import os no funcionaba con require() en TypeScript
--------------------------------------------------------------------------------

CUANDO OCURRIO:
  Al agregar el endpoint /whoami al BFF para mostrar el hostname del contenedor.

CODIGO ERRONEO:
  app.get('/whoami', (_req, res) => {
    const os = require('os')  // ERROR: require dinamico en modulo TypeScript
    res.json({ hostname: os.hostname() })
  })

MENSAJE DE ERROR:
  TypeError: require is not defined in ES module scope

CAUSA RAIZ:
  El proyecto usa TypeScript compilado a CommonJS. Los require() dinamicos
  dentro de funciones arrow pueden causar conflictos dependiendo del contexto
  de modulo. La forma correcta en TypeScript es el import estatico al inicio.

SOLUCION:
  Agregar al inicio del archivo bff-gateway/src/app.ts:
    import os from 'os'

  Y en el handler:
    app.get('/whoami', (_req, res) => {
      res.json({ hostname: os.hostname(), pid: process.pid, ... })
    })

LECCION:
  En TypeScript/Node.js, siempre usar import estatico en la parte superior
  del archivo. Los require() dinamicos son un patron de CommonJS puro
  que puede causar conflictos en proyectos con TypeScript.

--------------------------------------------------------------------------------
ERROR 9: Conflicto de puertos al escalar el BFF con docker compose --scale
--------------------------------------------------------------------------------

CUANDO OCURRIO:
  Al intentar ejecutar docker compose up --scale bff=3 con el puerto 3000
  mapeado en docker-compose.yml.

MENSAJE DE ERROR:
  Error response from daemon: driver failed programming external connectivity
  on endpoint bff-2: Bind for 0.0.0.0:3000 failed: port is already allocated

CAUSA RAIZ:
  El docker-compose.yml tenia:
    bff:
      ports:
        - "3000:3000"
  Cuando Docker intenta crear bff-2, quiere mapear el puerto 3000 del host
  al contenedor, pero ese puerto ya lo tiene bff-1. Un puerto del host solo
  puede asignarse a UN contenedor a la vez.

SOLUCION:
  Eliminar el mapeo de puertos del BFF en docker-compose.yml.
  Traefik (que si tiene el puerto 80 mapeado) actua como punto de entrada
  y envia el trafico internamente a http://bff:3000 sin necesidad de exponer
  el puerto 3000 al host.

  ANTES:
    bff:
      ports:
        - "3000:3000"

  DESPUES:
    bff:
      # Sin ports: Traefik enruta via red interna

LECCION:
  Si un servicio esta detras de un reverse proxy (Traefik), no necesita
  exponer puertos al host. El proxy es el unico que necesita el puerto expuesto.

4.2 EXPLICACION PROFUNDA DEL STACK TECNOLOGICO
================================================================================

Esta seccion explica QUE hace cada tecnologia, POR QUE fue elegida, y como
interactua con el resto del sistema.

--------------------------------------------------------------------------------
NODE.JS v22 LTS - RUNTIME
--------------------------------------------------------------------------------

Node.js es el entorno de ejecucion de JavaScript en el servidor.
Usa un modelo de I/O no bloqueante basado en eventos (event loop).

QUE SIGNIFICA NO BLOQUEANTE:
  En un servidor tradicional (Java/PHP con hilos), si 100 usuarios hacen
  peticiones a la BD simultaneamente, el servidor crea 100 hilos esperando
  la respuesta. Cada hilo consume memoria aunque solo este esperando.

  En Node.js, cuando se hace una consulta a la BD, el hilo principal NO espera.
  Registra un "callback" (o usa async/await) y sigue procesando otras peticiones.
  Cuando la BD responde, retoma esa peticion. Un solo hilo puede manejar miles
  de peticiones concurrentes de I/O.

POR QUE v22 LTS (Long Term Support):
  - Estabilidad garantizada con soporte hasta 2027
  - Incluye mejoras de performance del motor V8
  - Compatible con todos los paquetes usados en el proyecto

--------------------------------------------------------------------------------
TYPESCRIPT v5 - LENGUAJE
--------------------------------------------------------------------------------

TypeScript es un superset de JavaScript que agrega tipado estatico opcional.
Se compila a JavaScript antes de ejecutarse.

QUE AGREGA AL PROYECTO:
  Sin TypeScript:
    function crearProyecto(datos) { ... }  // datos puede ser cualquier cosa

  Con TypeScript:
    interface CrearProyectoDto {
      name: string
      description: string
    }
    function crearProyecto(datos: CrearProyectoDto) { ... }
    // El compilador avisa si se pasa un campo mal escrito o faltante

COMO SE USA EN INNOVATECH:
  - Los DTOs (Data Transfer Objects) definen que campos acepta cada endpoint
  - Prisma genera tipos TypeScript automaticamente desde el schema
  - Los errores de tipo se detectan en tiempo de compilacion (npm run build)
    no en produccion cuando un usuario ya recibio un error

TSCONFIG CRITICO:
  "module": "commonjs" y "target": "es2020" son criticos.
  Si se usa "module": "nodenext", los imports cambian de sintaxis y Prisma
  tiene problemas de compatibilidad.

--------------------------------------------------------------------------------
EXPRESS v4 - FRAMEWORK WEB
--------------------------------------------------------------------------------

Express es un framework minimalista para crear servidores HTTP en Node.js.
Funciona con un sistema de middlewares: funciones que procesan la peticion
en cadena antes de llegar al handler final.

FLUJO DE UNA PETICION EN EXPRESS:
  Peticion HTTP entrante
    -> CORS middleware (verifica que el origen este permitido)
    -> JSON parser middleware (convierte el body de texto a objeto JS)
    -> Auth middleware (verifica el JWT)
    -> Route handler (logica del endpoint)
    -> Respuesta HTTP

EN INNOVATECH:
  - Cada microservicio (ms-tasks, ms-projects, ms-teams) tiene su propio servidor
    Express con su propio conjunto de middlewares
  - El BFF tiene un Express adicional que actua como orquestador
  - El orden de los middlewares es critico: el health check (/health) se registra
    ANTES del middleware JWT para que sea accesible sin token

--------------------------------------------------------------------------------
PRISMA ORM v5.22.0 - CAPA DE BASE DE DATOS
--------------------------------------------------------------------------------

Prisma es un ORM (Object Relational Mapper) que permite trabajar con la BD
usando objetos TypeScript en lugar de SQL crudo.

SIN PRISMA (SQL crudo):
  const result = await db.query(
    "INSERT INTO tasks (id, name, status, project_id) VALUES ($1, $2, $3, $4)",
    [uuid(), name, 'PENDING', projectId]
  )
  // Sin tipos, sin autocompletado, error en runtime si el nombre de campo cambia

CON PRISMA:
  const task = await prisma.task.create({
    data: { name, status: 'PENDING', projectId }
  })
  // Con tipos, con autocompletado, error en compilacion si el campo no existe

COMO FUNCIONA EN INNOVATECH:
  1. El schema.prisma define el modelo de datos (equivale al DDL de SQL)
  2. npx prisma generate crea el cliente TypeScript con los tipos del schema
  3. npx prisma migrate deploy crea/modifica las tablas en PostgreSQL
  4. El codigo usa el cliente para hacer queries con tipos

POR QUE VERSION FIJA 5.22.0 (sin ^):
  Prisma 6 cambio la forma de manejar los enums. Si se instalara v6
  automaticamente, el schema.prisma y las queries fallarian con errores
  oscuros. La version fija garantiza que todos los desarrolladores usen
  la misma version.

--------------------------------------------------------------------------------
POSTGRESQL v15 - BASE DE DATOS
--------------------------------------------------------------------------------

PostgreSQL es una base de datos relacional de codigo abierto.
En Innovatech cada microservicio tiene su PROPIA instancia de PostgreSQL
(Database per Service pattern).

POR QUE UNA BD POR MICROSERVICIO:
  Si los tres microservicios compartieran una sola BD:
  - Un cambio de schema en ms-tasks podria romper ms-projects
  - Un microservicio caido que bloquea la BD afectaria a todos
  - No se pueden escalar de forma independiente

  Con BD independientes:
  - Cada microservicio es completamente autonomo
  - Pueden tener schemas diferentes sin interferencia
  - Se pueden escalar, hacer backup y migrar de forma independiente

DATOS EN EL SISTEMA:
  db-tasks    (puerto 5433): tabla Task con UUID, name, status, projectId...
  db-projects (puerto 5434): tabla Project con UUID, name, status...
  db-teams    (puerto 5435): tabla Team con UUID, name, status, members[]

--------------------------------------------------------------------------------
JWT (JSON WEB TOKEN) - AUTENTICACION
--------------------------------------------------------------------------------

JWT es un estandar para transmitir informacion de autenticacion como un token
firmado criptograficamente.

ESTRUCTURA DEL TOKEN:
  eyJhbGciOiJIUzI1NiJ9.eyJ1c2VybmFtZSI6ImFkbWluIn0.firma

  Parte 1 (header):  algoritmo de firma (HS256)
  Parte 2 (payload): datos del usuario (username, role, expiracion)
  Parte 3 (firma):   hash del header+payload con JWT_SECRET

POR QUE ES SEGURO:
  La firma se calcula con HMAC-SHA256 usando JWT_SECRET.
  Si alguien modifica el payload (ej: cambia role a "admin"), la firma
  ya no coincide y el servidor lo rechaza con 403.
  Sin conocer JWT_SECRET, es computacionalmente imposible generar una firma valida.

ZERO TRUST EN INNOVATECH:
  Cada microservicio tiene su propio middleware que valida el JWT.
  Cuando el BFF llama a ms-tasks, le pasa el token del usuario original.
  ms-tasks valida ese token contra su propio JWT_SECRET (que es el mismo).
  Esto garantiza que incluso si alguien accediera directamente a ms-tasks
  saltandose el BFF, igual necesita un JWT valido.

FLUJO DE AUTENTICACION:
  1. Cliente: POST /auth/login con username + password
  2. BFF: bcrypt.compare(password, hash) -> si ok, crea JWT firmado
  3. Cliente: guarda el token y lo envia en cada peticion como:
     Authorization: Bearer eyJhbG...
  4. Cada servicio: verifica la firma, extrae el payload, permite o rechaza

--------------------------------------------------------------------------------
BCRYPT - HASHING DE PASSWORDS
--------------------------------------------------------------------------------

bcrypt es un algoritmo de hashing disenado especificamente para passwords.

POR QUE NO SHA256 O MD5:
  SHA256 y MD5 son rapidos (millones de hashes por segundo en una GPU).
  Eso los hace vulnerables a ataques de diccionario o fuerza bruta.

  bcrypt es deliberadamente lento. Con saltRounds=10, tarda ~100ms por hash.
  Un atacante que roba los hashes tardaria anos en probar todas las combinaciones.

COMO FUNCIONA:
  Al guardar el password:
    bcrypt.hashSync('admin123', 10) -> "$2a$10$xyz...abc" (hash unico)
  Al verificar:
    bcrypt.compare('admin123', '$2a$10$xyz...abc') -> true
  Mismo password, hashes distintos cada vez (por el "salt" aleatorio interno).

EN INNOVATECH:
  Los hashes se calculan al inicializar el modulo auth.routes.ts (una sola vez).
  No se almacenan en BD, estan en memoria del proceso BFF.
  En produccion real estarian en una BD con hashes precalculados.

--------------------------------------------------------------------------------
OPOSSUM - CIRCUIT BREAKER
--------------------------------------------------------------------------------

Circuit Breaker es un patron de diseno que protege al sistema cuando un
microservicio falla repetidamente.

ANALOGIA CON EL DISYUNTOR ELECTRICO:
  En electricidad: si hay un cortocircuito, el disyuntor "abre" y corta la
  corriente para evitar danos. Se puede "resetear" manualmente.

  En software: si ms-tasks falla 50% de las veces, el circuit breaker "abre"
  y retorna un fallback inmediato sin intentar llamar al servicio caido.
  Despues de 10 segundos, prueba una llamada (half-open). Si pasa, cierra.

ESTADOS EN INNOVATECH:
  CLOSED   -> Normal. Todas las llamadas van a ms-tasks.
  OPEN     -> ms-tasks fallo demasiado. Las llamadas retornan el fallback
              inmediatamente sin esperar timeout.
  HALF-OPEN-> 10s despues de abrirse, deja pasar UNA llamada de prueba.

CONFIGURACION EN EL BFF:
  timeout: 3000              // Llamada que tarda mas de 3s cuenta como fallo
  errorThresholdPercentage: 50  // Si 50% fallan, abre el circuito
  resetTimeout: 10000           // 10s en OPEN antes de pasar a HALF-OPEN

POR QUE ES IMPORTANTE:
  Sin circuit breaker: si ms-tasks esta caido, cada peticion al BFF espera 3s
  de timeout antes de fallar. 100 usuarios = 100 hilos bloqueados 3s cada uno.
  El BFF se vuelve lento y puede saturarse.

  Con circuit breaker: despues de los primeros fallos, el circuito abre y
  las respuestas son inmediatas (~1ms de fallback). El BFF sigue respondiendo
  normal aunque ms-tasks este completamente caido.

--------------------------------------------------------------------------------
TRAEFIK v3.0 - LOAD BALANCER / REVERSE PROXY
--------------------------------------------------------------------------------

Traefik es un reverse proxy y load balancer moderno escrito en Go.

DIFERENCIA ENTRE REVERSE PROXY Y LOAD BALANCER:
  Reverse proxy: recibe peticiones externas y las reenvía a servicios internos.
    El cliente no sabe la IP real del servicio interno.
  Load balancer: distribuye las peticiones entre multiples instancias de un
    servicio para no sobrecargar ninguna.
  Traefik hace ambas cosas.

COMO FUNCIONA EN INNOVATECH:
  1. Cliente hace GET http://localhost/api/projects
  2. Traefik recibe en puerto 80
  3. Router "bff" matchea PathPrefix("/")
  4. Load balancer selecciona el siguiente backend en round-robin
  5. Traefik reenvía la peticion a http://172.18.0.X:3000/api/projects
  6. BFF responde y Traefik devuelve la respuesta al cliente

HEALTH CHECK DE TRAEFIK:
  Cada 10 segundos, Traefik hace GET /health a cada backend.
  Si un backend responde distinto de 2xx, Traefik lo saca de la rotacion.
  El cliente nunca recibe un error porque se llego a un backend caido.

FILE PROVIDER vs DOCKER PROVIDER:
  Docker provider: Traefik pregunta al daemon Docker que contenedores hay
    y extrae la config de los labels Docker. Ideal en Linux nativo.
    No funciona en Windows Docker Desktop (problema de socket).
  File provider: Traefik lee un archivo YAML estatico con la configuracion.
    Requiere reinicio o file watcher para actualizarse. Funciona en Windows.

--------------------------------------------------------------------------------
DOCKER Y DOCKER COMPOSE v3.8
--------------------------------------------------------------------------------

Docker: tecnologia de contenedores que empaqueta una aplicacion con todas sus
dependencias en una imagen portable que corre igual en cualquier maquina.

DIFERENCIA CONTENEDOR vs MAQUINA VIRTUAL:
  VM: emula hardware completo (CPU, RAM, disco). Cada VM tiene su propio OS.
    Arranca en minutos. Ocupa GB de RAM.
  Contenedor: comparte el kernel del OS del host. Solo empaqueta la aplicacion
    y sus librerias. Arranca en segundos. Ocupa MB de RAM.

DOCKERFILE EN INNOVATECH:
  FROM node:22-alpine  -> imagen base (Node 22 en Alpine Linux ~50MB)
  COPY package*.json   -> copia solo el manifest de dependencias
  RUN npm install      -> instala dependencias (se cachea si package.json no cambio)
  COPY . .             -> copia el codigo fuente
  RUN npm run build    -> compila TypeScript a JavaScript
  HEALTHCHECK          -> define como Docker verifica si el contenedor esta sano
  CMD ["node", "dist/app.js"]  -> comando que arranca la aplicacion

DOCKER COMPOSE:
  Herramienta para definir y correr multi-contenedor en un archivo YAML.
  En Innovatech, levanta 10 servicios con un solo comando (docker compose up).
  Maneja el orden de arranque (depends_on), redes internas y variables de entorno.

--------------------------------------------------------------------------------
DOCKER SWARM - ORQUESTADOR
--------------------------------------------------------------------------------

Docker Swarm convierte uno o mas hosts Docker en un cluster gestionado.
En Innovatech se usa para auto-healing y replicas del BFF.

CONCEPTOS CLAVE:
  Stack:   grupo de servicios desplegados juntos (docker stack deploy)
  Service: definicion de como corre un contenedor con cuantas replicas
  Task:    instancia concreta de un service (un contenedor en ejecucion)
  Manager: nodo que toma decisiones (en Innovatech, el unico nodo)
  Overlay: red virtual que conecta contenedores en el cluster

AUTO-HEALING:
  El manager monitorea el estado de cada task.
  Si una task falla (exit code != 0 o HEALTHCHECK failing), el manager
  la marca como Failed y crea una nueva task automaticamente.
  El proceso tarda entre 5 y 15 segundos dependiendo de restart_policy.delay.

ROLLING UPDATE:
  Al actualizar una imagen (docker service update --image), Swarm reemplaza
  replicas de a una (parallelism: 1). Espera que la nueva este healthy antes
  de apagar la anterior (order: start-first). Si falla, hace rollback
  automatico (failure_action: rollback).

DIFERENCIA CON KUBERNETES:
  Kubernetes es mas potente (auto-scaling basado en CPU, namespaces, RBAC,
  Ingress resources, etc.) pero es mucho mas complejo de instalar y operar.
  Docker Swarm es parte de Docker y no requiere instalacion adicional.
  Para este proyecto, Swarm es la eleccion correcta en complejidad/beneficio.

================================================================================
4.3 DEFENSA TECNICA DEL STACK (PARA EVALUACION ORAL)
================================================================================

Esta seccion responde preguntas tipicas de una defensa tecnica oral.
Las respuestas estan redactadas para ser explicadas en voz alta.

--------------------------------------------------------------------------------
"¿Por que microservicios y no una aplicacion monolitica?"
--------------------------------------------------------------------------------

Un monolito es mas simple de desarrollar al inicio, pero tiene limitaciones:
- Si una parte falla (ej: modulo de tareas), toda la aplicacion cae
- Si una parte necesita mas recursos, hay que escalar todo el sistema
- Equipos distintos no pueden trabajar en paralelo sin conflictos

Con microservicios en Innovatech:
- ms-tasks puede caer y ms-projects y ms-teams siguen funcionando
- El BFF protege al usuario con el circuit breaker (fallback inmediato)
- Se puede escalar solo el BFF (que es el cuello de botella de trafico)
  sin tocar las bases de datos ni los microservicios

La arquitectura actual refleja el dominio del negocio: Proyectos, Tareas y
Equipos son dominios distintos con sus propios datos y reglas de negocio.
Cada uno tiene su propia base de datos, lo que garantiza bajo acoplamiento.

--------------------------------------------------------------------------------
"¿Por que JWT y no sesiones?"
--------------------------------------------------------------------------------

Las sesiones tradicionales almacenan el estado en el servidor (en memoria o BD).
Eso significa que todas las peticiones deben ir al mismo servidor o sincronizar
el estado entre servidores, lo que es incompatible con el escalado horizontal.

JWT es stateless: toda la informacion esta en el token firmado.
Cualquiera de las 3 replicas del BFF puede verificar el token sin consultar
ningun almacen centralizado. El usuario puede ir al BFF-1 en una peticion y
al BFF-3 en la siguiente, y ambas verifican el mismo token correctamente.

El modelo Zero Trust agrega una capa extra: aunque el BFF ya verifico el JWT,
cada microservicio lo verifica de nuevo. Esto significa que si alguien
comprometiera el BFF y le quitara la capa de autenticacion, igual no podria
acceder a los microservicios. La seguridad no depende de un solo punto.

--------------------------------------------------------------------------------
"¿Por que Traefik y no NGINX?"
--------------------------------------------------------------------------------

NGINX es excelente como reverse proxy pero tiene limitaciones para entornos
dinamicos: su configuracion es estatica y requiere recarga manual cuando
se agregan o quitan backends.

Traefik fue disenado para entornos de contenedores donde los servicios
aparecen y desaparecen dinamicamente. Tiene:
- Dashboard visual en tiempo real (localhost:8080)
- Health checks automaticos integrados
- Soporte nativo para Docker, Swarm y Kubernetes
- Recarga de configuracion sin downtime

En el proyecto usamos el file provider (en lugar del Docker provider)
debido a una limitacion de Windows Docker Desktop con el socket de Docker.
Esta es una solucion valida de produccion: muchas empresas usan file provider
cuando el entorno no es compatible con el autodescubrimiento dinamico.

--------------------------------------------------------------------------------
"¿Que pasa si el BFF Gateway se cae?"
--------------------------------------------------------------------------------

Antes de implementar el load balancer y Swarm:
El BFF era un punto unico de fallo. Si caia, todo el sistema quedaba
inaccesible para los clientes.

Despues de la implementacion actual:
1. Traefik tiene health checks cada 10 segundos sobre /health del BFF
2. Si una replica del BFF cae, Traefik la saca de la rotacion automaticamente
3. Docker Swarm detecta la replica caida y levanta una nueva en ~10 segundos
4. Durante esos 10 segundos, las otras 2 replicas siguen atendiendo trafico
5. El cliente nunca recibe un error: Traefik simplemente no le manda mas
   peticiones a la replica problematica

Esto elimina el punto unico de fallo que existia antes.

--------------------------------------------------------------------------------
"¿Como funciona el Circuit Breaker con las 3 replicas del BFF?"
--------------------------------------------------------------------------------

Cada replica del BFF tiene su PROPIO estado de circuit breaker en memoria.
Cuando la replica A detecta que ms-tasks falla y abre su circuito, las
replicas B y C tienen sus circuitos aun cerrados (no saben lo que le paso a A).

Esto tiene una implicacion practica: si un cliente llega a la replica A,
recibe el fallback inmediatamente. Si llega a la replica B, espera el timeout
de 3 segundos antes de activar su propio circuito.

En produccion con muchos clientes, esto funciona bien porque rapidamente
todas las replicas detectan el fallo y abren sus circuitos. En una demo
con pocas peticiones, puede verse comportamiento inconsistente dependiendo
de a que replica llega cada peticion.

--------------------------------------------------------------------------------
"¿Por que bcrypt y no guardar el password en texto plano o con MD5?"
--------------------------------------------------------------------------------

Texto plano: si alguien accede a la BD (por SQL injection o backup robado),
ve todos los passwords directamente. Los usuarios suelen reutilizar passwords,
por lo que un atacante podria acceder a sus cuentas en otros servicios.

MD5/SHA1: aunque no se ven directamente, existen "rainbow tables" (tablas
precomputadas de MD5/SHA1 de millones de passwords comunes). El atacante
no necesita calcular nada: solo busca el hash en la tabla.

bcrypt tiene dos ventajas criticas:
1. Salt automatico: cada hash incluye un valor aleatorio (el "salt"). El mismo
   password produce hashes distintos cada vez, haciendo inutiles las rainbow tables.
2. Work factor ajustable: saltRounds=10 hace que calcular UN hash tome ~100ms.
   Si el atacante quiere probar 1 millon de passwords, tardaria 27 horas
   con hardware especializado. Con MD5 tardaria segundos.

--------------------------------------------------------------------------------
"¿Que es el patron Database per Service y por que lo usamos?"
--------------------------------------------------------------------------------

Database per Service significa que cada microservicio tiene su propia base de datos
que ningun otro microservicio puede acceder directamente.

Si ms-projects necesita informacion de ms-tasks (para verificar si hay tareas
pendientes antes de completar un proyecto), NO consulta la BD de ms-tasks
directamente. En cambio, le hace una peticion HTTP a ms-tasks con su JWT.

Ventajas en Innovatech:
1. Si la BD de ms-tasks falla, ms-projects y ms-teams siguen funcionando
2. Podriamos cambiar ms-tasks de PostgreSQL a MongoDB sin afectar a nadie
3. Las migraciones de schema de ms-tasks no pueden romper ms-projects
4. Cada BD puede estar en un servidor distinto si necesitamos mas capacidad

La "desventaja" es que no se pueden hacer JOINs entre tablas de distintos
microservicios. En Innovatech esto se resuelve: la logica que necesita datos
de varios servicios vive en el BFF, que orquesta las llamadas necesarias.

================================================================================
4.4 FLUJO COMPLETO DE DATOS (DE EXTREMO A EXTREMO)
================================================================================

Esta seccion traza el recorrido exacto que hace una peticion real desde el
navegador hasta la base de datos y de regreso.

CASO DE USO: Usuario hace login y crea un proyecto.

--------------------------------------------------------------------------------
FASE 1: LOGIN (POST /auth/login)
--------------------------------------------------------------------------------

PASO 1: El navegador (frontend React en puerto 5173) ejecuta:
  axios.post('http://localhost/auth/login', { username: 'admin', password: 'admin123' })
  La URL apunta a http://localhost (Traefik en puerto 80).
  El valor viene de la variable de entorno VITE_BFF_URL.

PASO 2: La peticion TCP llega al puerto 80 del host.
  Docker redirige ese puerto al contenedor de Traefik.

PASO 3: Traefik recibe la peticion en su entrypoint "web" (puerto 80).
  El router "bff" matchea la regla PathPrefix("/") para TODAS las rutas.
  Traefik selecciona el siguiente servidor en round-robin del bff-service.
  Traefik reenvía la peticion via TCP al backend seleccionado (ej: 172.18.0.9:3000).

PASO 4: El contenedor BFF Gateway (replica 1) recibe la peticion.
  Express procesa los middlewares en orden:
    -> JSON parser: convierte el body de texto a { username: 'admin', password: 'admin123' }
    -> CORS: verifica que el origen (http://localhost:5173) esta permitido
    -> Router: la ruta POST /auth/login va al handler de auth.routes.ts
  auth.routes.ts:
    1. Extrae username y password del body
    2. Busca el usuario en el objeto USERS en memoria
    3. Llama a bcrypt.compare('admin123', hash_almacenado)
    4. bcrypt verifica criptograficamente que el password es correcto
    5. jwt.sign({ username, role, displayName }, JWT_SECRET, { expiresIn: '24h' })
    6. Genera un token firmado de 24 horas de validez
  El BFF responde 200 con el token.

PASO 5: Traefik recibe la respuesta del backend y la reenvía al navegador.

PASO 6: React recibe { token, role, username } y guarda el token.
  En bffClient.ts: axiosInstance.defaults.headers.common['Authorization'] = `Bearer ${token}`
  Todos los requests futuros incluiran este header automaticamente.

--------------------------------------------------------------------------------
FASE 2: CREAR PROYECTO (POST /api/projects)
--------------------------------------------------------------------------------

PASO 7: El navegador envia:
  POST http://localhost/api/projects
  Headers: { Authorization: "Bearer eyJhbGciOiJIUzI1NiJ9..." }
  Body: { name: "Proyecto Alpha", description: "Primer proyecto" }

PASO 8: Traefik recibe en puerto 80.
  El router "bff" matchea PathPrefix("/").
  Round-robin selecciona la replica 2 del BFF (172.18.0.10:3000).
  (Puede ser una replica diferente a la del login, esto es normal)

PASO 9: BFF Gateway replica 2 recibe la peticion.
  Express procesa middlewares:
    -> JSON parser: convierte body
    -> CORS: verifica origen
    -> Auth middleware (verifyToken):
       1. Lee el header Authorization: Bearer eyJhbG...
       2. Extrae el token del string (quita el prefijo "Bearer ")
       3. jwt.verify(token, JWT_SECRET) -> verifica la firma criptografica
       4. Si la firma no coincide: responde 403 Forbidden
       5. Si el token expiro: responde 403 Forbidden
       6. Si todo ok: pone el payload del token en req.user y llama next()
    -> Router: la ruta POST /api/projects va a projects.routes.ts en el BFF

PASO 10: projects.routes.ts en el BFF llama a projects.service.ts.
  projects.service.ts tiene un Circuit Breaker de Opossum para la llamada a
  ms-projects. El circuit breaker evalua su estado:
    - CLOSED: procede a llamar a ms-projects
    - OPEN: retorna inmediatamente con el fallback (no llama a ms-projects)
  Asumiendo CLOSED, el breaker ejecuta la funcion de llamada.

PASO 11: El BFF hace una peticion HTTP interna a ms-projects:
  axios.post('http://ms-projects:3002/projects', body, {
    headers: { Authorization: `Bearer ${req.headers.authorization.split(' ')[1]}` }
  })
  El BFF REENVÍA el JWT original del usuario. No crea uno nuevo.
  "ms-projects" resuelve via DNS interno de Docker a la IP del contenedor ms-projects.

PASO 12: ms-projects recibe la peticion.
  Express procesa middlewares:
    -> JSON parser
    -> /health esta registrado ANTES del auth middleware (es publico)
    -> Auth middleware (su propio verifyToken):
       jwt.verify(token, process.env.JWT_SECRET) -> verifica el mismo secret
       Si el BFF enviara una peticion sin token o con token invalido, ms-projects
       la rechazaria con 401/403. Esto es el Zero Trust.
    -> Router: POST /projects va a project.controller.ts

PASO 13: project.controller.ts valida los datos:
  if (!name) return res.status(400).json({ error: 'El nombre es obligatorio' })
  if (!description) return res.status(400).json({ error: 'La descripcion es obligatoria' })
  Si todo ok, llama a project.service.ts con los datos validados.

PASO 14: project.service.ts llama a Prisma:
  const project = await prisma.project.create({
    data: { name, description, status: 'PLANNED' }
  })
  Prisma genera el SQL:
    INSERT INTO "Project" (id, name, description, status, "createdAt", "updatedAt")
    VALUES ('uuid-generado', 'Proyecto Alpha', 'Primer proyecto', 'PLANNED', now(), now())
  Envia el SQL a PostgreSQL (contenedor db-projects via red Docker interna).

PASO 15: PostgreSQL ejecuta el INSERT y retorna la fila creada.
  Prisma convierte la fila a objeto TypeScript: { id: '...', name: '...', status: 'PLANNED', ... }

PASO 16: La respuesta sube por la cadena de vuelta:
  project.service.ts -> project.controller.ts -> Express (res.status(201).json(project))
  -> ms-projects responde HTTP 201 con el proyecto creado en JSON

PASO 17: El BFF recibe la respuesta de ms-projects via axios.
  El circuit breaker registra la llamada como exitosa (no incrementa el contador de fallos).
  El BFF responde al cliente con el mismo 201 + body.

PASO 18: Traefik recibe la respuesta del BFF y la reenvía al navegador.

PASO 19: React recibe el proyecto creado y actualiza el estado de la aplicacion.
  La UI muestra el nuevo proyecto en la lista con status PLANNED.

--------------------------------------------------------------------------------
